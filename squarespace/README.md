# Fletschhorn → Squarespace (paste-ready)

This folder is the **cut-and-paste version** of the site. The prototype files in
the repo root are standalone HTML documents and **cannot** be pasted into
Squarespace as-is (they have `<!doctype>/<html>/<head>/<body>` wrappers, a dead
`<link>` to `assets/fh-global.css`, GitHub-Pages links, and a header/footer that
self-loads by `fetch()`). Everything here has been reworked so each piece drops
straight into the matching Squarespace slot.

Originals in the repo root are untouched. Everything here is regenerated by
`_build/build_squarespace.py` (re-run it after editing an original).

```
squarespace/
├── global/
│   ├── 1-custom-css.css                  → Design ▸ Custom CSS
│   ├── 2-code-injection-header.html      → Settings ▸ Advanced ▸ Code Injection ▸ HEADER
│   └── 3-code-injection-footer.html      → Settings ▸ Advanced ▸ Code Injection ▸ FOOTER
├── pages/                                (one file per page; each has 2 paste targets)
│   ├── home.html            (Home,  slug "/")
│   ├── property.html        (slug "/property")
│   ├── experiences.html     (slug "/experiences")
│   ├── location.html        (slug "/location")
│   ├── wedding-event.html   (slug "/wedding-event")
│   ├── about-us.html        (slug "/about-us")
│   ├── booking.html         (slug "/booking")
│   ├── checkout.html        (slug "/checkout")
│   └── confirmation.html    (slug "/confirmation")
└── _build/build_squarespace.py           (the generator — see “Rebuilding”)
```

## Step 1 — Global shell (do this once)

These three apply to every page, so paste them once at the site level.

1. **Custom CSS** — open `global/1-custom-css.css`, copy everything, paste into
   **Design ▸ Custom CSS**.
2. **Header injection** — open `global/2-code-injection-header.html`, copy
   everything, paste into **Settings ▸ Advanced ▸ Code Injection ▸ HEADER**.
   This is the site-wide top nav, mobile menu, language switch, mobile footbar
   and the booking-date toolbar.
3. **Footer injection** — open `global/3-code-injection-footer.html`, copy
   everything, paste into **Settings ▸ Advanced ▸ Code Injection ▸ FOOTER**.
   This is the site-wide footer; it also hides Squarespace’s native footer.

### ⚙️ Make the nav links match your page URLs
Near the top of `global/2-code-injection-header.html` there is an editable map:

```js
const FH_PAGES = {
  home:        "/",
  property:    "/property",
  experiences: "/experiences",
  location:    "/location",
  events:      "/wedding-event",
  about:       "/about-us",
  booking:     "/booking"
};
```

These must equal the **real URL slugs** of your Squarespace pages. If you create
a page at a different slug (e.g. `/the-property`), change the value here. The
footer links (`global/3-code-injection-footer.html`) use the same slugs — keep
the two in sync.

## Step 2 — One page at a time

For each file in `pages/`, create a Squarespace page whose **URL slug matches the
table above**, then:

- **PASTE 1 → Page settings ▸ Advanced ▸ Page Header Code Injection.**
  SEO meta tags + structured data (`JSON-LD`). *Optional* — if you’d rather set
  titles/descriptions in Squarespace’s built-in **SEO** panel, you can skip
  PASTE 1. Structured data (the `application/ld+json` block) is worth keeping for
  rich results.
- **PASTE 2 → Edit the page ▸ add a Code block ▸ paste everything below the
  PASTE 2 marker.** This is the page’s styles + markup + scripts. The header and
  footer are **not** in here — they come from Step 1.

Each file is clearly split with `▌ PASTE 1` / `▌ PASTE 2` comment markers.

> **Big pages (experiences, about-us, home).** These code blocks are large. If
> the Code-block editor feels sluggish, paste PASTE 2 into that page’s
> **Page Footer Code Injection** instead of a Code block — it renders the same
> because the markup mounts itself.

## What was changed vs. the prototype

| Area | Prototype | Squarespace version |
|------|-----------|---------------------|
| Page wrappers | full `<!doctype><html><head><body>` docs | wrappers stripped; only the page block remains |
| Global CSS | `<link href="assets/fh-global.css">` on every page | dead link removed; CSS lives in Custom CSS |
| Header/footer | each page `fetch()`-loaded `Header.html`/`footer.html` into `#fh-header-slot` / `#fh-footer-slot` | slots removed (loader no-ops safely); header/footer come from global Code Injection |
| Header timing | ran only after the fetch, post-DOM | wrapped in a DOM-ready guard so it works injected into `<head>` |
| Nav base URL | `https://lschacht99.github.io/Fletschhorn-berry/…` | editable root-relative `FH_PAGES` slug map |
| Internal links | `Property.html`, `booking.html?…`, etc. | rewritten to slugs (`/property`, `/booking?…`, the full booking→checkout→confirmation flow) |
| SEO/meta | inside each `<head>` | split into PASTE 1 for per-page code injection |
| Performance | — | `preconnect`/`dns-prefetch` added for Guesty, Cloudinary, ImageKit, Saas-Fee CDNs |

## Things to check / caveats

- **Slugs must match.** The nav, footer and the booking→checkout→confirmation
  hand-off all rely on the slugs above. If a page lives at a different URL,
  update `FH_PAGES` (header) and the matching footer link.
- **Social image.** `og:image` points to
  `https://www.fletschhorn.ch/assets/og-image.jpg`. Make sure that file exists on
  the live domain (or set the Social image in Squarespace’s SEO panel).
- **Booking backend.** Live availability/booking calls the Guesty Cloudflare
  Worker (`fletschhorn-guesty-api.bookings-e2d.workers.dev`). That keeps working
  unchanged; no action needed.
- **Canonicals** use `https://www.fletschhorn.ch/…`. If your Squarespace site is
  on a different/staging domain, Squarespace generates its own canonicals — the
  PASTE 1 canonicals are only correct once you’re on the production domain.
- **`Test.html`** in the repo root is a scratch/prototype page and is **not**
  included here.
- Languages (EN/FR/DE/RU/HE incl. RTL) are handled in-page by the existing
  scripts — nothing extra to configure.

## Rebuilding

If you edit an original in the repo root, regenerate this folder:

```bash
python3 squarespace/_build/build_squarespace.py
```

The script never touches the originals; it only rewrites `squarespace/global/`
and `squarespace/pages/`. Page slugs live in the `SLUGS` map at the top of the
script if you want different defaults.
